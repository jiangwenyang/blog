## React 中的 Reconciliation

当 React 元素需要重新渲染的时候，需要将一棵树转换为另外一棵树。通过对比算法来解决如何最小化操作转换，目前最先进的算法的时间复杂度是 O(n<sup>3</sup>)。
如果进行完全比较的话性能会很差，React 的 Diff 算法做了两点假设，实现了启发的 O(n)算法：

1. 两个不同类型的元素直接产生不同的树
2. 通过 keys 属性标志元素

###Diff 算法
比较的时候首先看根节点，分为了以下三种情况

#### 1.不同类型的元素

两个元素不同类型有以下几种情况：

- 都为 DOM 元素，但是不是同一种标签,例如`<a>`和`<p>`
- 都为组件元素,但是不是同一个组件，例如`<Comments>`和`<Posts>`
- DOM 元素和组件元素，例如`<div>`和`<Comments>`

当两个元素不是同类型的时候，React 将会直接写在掉旧的 DOM 节点，然后重新渲染新的节点，就好像旧的节点不存在一样，所有的旧节点状态被直接销毁。

#### 2.相同类型的 DOM 元素

当两个元素都为 DOM 元素，并且为同种标签，例如都为`<div>`时，React 将会进行保持相同的根节点，并只更新改变的属性，并且会对 style 属性进行 diff。然后递归比较子元素。

#### 3.相同类型的组件元素

当两个组件元素相同时，例如都为`<Post>`，则进行更新的时候旧组件的实例会成为新组件的实例，然后只是更新状态，即不会发生`componentWillUnmount()`,只会调用`componentWillReceiveProps()`和`componentWillUpdate()`方法来实现更新，就好像是在同一个组件上更新了 state 一样。

#### 递归子节点优化

当 React 递归 DOM 子节点的时候，例如在 ul 中插入了一个 li，则 React 会去比较所有的 li。有以下三种情况：

1. 插入到已有 li 的最前面
2. 插入到已有 li 的中间
3. 插入到已有 li 的最后

当发生第三种的时候 React 会比较所有节点，并且发现前面的都没有改变，只是在最后添加了一个，然后就会保留前面的，只在最后的时候添加一个。
当发生前面两种的时候，React 并不知道你是添加了一个，而会通过比较原来的 li 和新的 li 的每一项，比如原来的第一个 li 和新的第一个 li，原来的第二个 li 和新的第二个 li，发现都不一样。最后便会将所有的 li 重新渲染一遍。
**Keys**
解决办法就是在渲染的时候手动给 li 添加一个 key 属性，然后 React 会对 key 属性进行匹配，性能更好。

> - key 只需要在兄弟节点中唯一，不需要全局唯一
> - 一般可以使用数据的 id 或者使用索引作为 key（如果返回的数据重新排序，会导致变慢）
